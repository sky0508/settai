# 接待ナビ（settai-navi） — 要件定義書（spec）

> このドキュメントは「何を・なぜ」を定義する。「どう作るか」は `design.md`（設計）と `plan.md`（実装計画）に分離する。

## 1. 背景・課題

接待の店選びは「暗黙知の失敗」が起きやすい領域。取り返しがつかないぶん、失敗コストが高い。主な痛点は2つ:

1. **相手企業ごとの制約（NG）** — 例: キリン相手の接待では、競合（アサヒ / サントリー / サッポロ）のビールしか出さない店は失礼にあたる。ところが「この店が何系のビールを出すか」という情報は、Google Places にも食べログにも構造化データとして存在しない。ベテランの頭の中にしかない。
2. **新人にセンスがない** — 格式・予算・相手の好み・過去に行った店を踏まえた判断ができず、知らずに失礼をしてしまう。

## 2. 目的・ゴール

### 位置づけ（重要）

**売るプロダクト候補**（将来 SaaS）。単なる社内ツールではない。この位置づけが製品の核を決める:

- 格式/個室/予算/NG食材の「基礎判定」は、食べログ＋できる人間で既に足りる → **差別化にならない = table-stakes（前提機能）**。
- プロダクトが存在していい理由（moat）は、そこではなく **暗黙知の自動化と社内ナレッジの側** にある。

### 核（賭けどころ）

- **入口価値**: **相手企業ごとの「暗黙の地雷」を根拠つきで回避する**。競合製品NG（例: キリン相手にアサヒ専売店）はその最も分かりやすい **第1インスタンス**であって、看板は「ビールチェッカー」ではなく「相手企業ごとの地雷を当てる」汎用フレーム。
- ==**継続価値（moat）**: 接待記録に評価とシチュエーションを紐づけて蓄積し、**社内の接待ナレッジ共有基盤**（使うほど賢くなるデータ資産）へ育てる。==
- **table-stakes**: 格式・個室・予算・相手のNG食材を満たした「失礼しない店」を根拠つきで提示する（新人が納得して選べる状態）。

> ★ wedge/moat の仮置き（要 Sora 確認）: 「入口＝地雷回避／moat＝社内ナレッジ」。ここが変わると v1 の中身が再分岐する。

### 検証ファースト（作る前に殺す）

売り物である以上、SaaS を建てる前に **上流の不確実性を安く潰す**:
0. **そもそも効く軸・重みが未確定**（現行 spec は個室有無と競合ビールしか拾えていない）→ **センスの良い社員へ暗黙知ヒアリングで軸を要件化**（叩き台＝status.md の「暗黙知9軸 A〜I」）。特に高価値な A(格式の釣り合い)・B(個室の質)・C(接待オペ適性) は Places に無く、手タグ・相手プロフィール・社内記録でしか埋まらない。
1. **暗黙知は使える精度で自動化できるか**（enrichment 再現性 → §8・§9 の spike）。
2. **痛みは本物で、金を払うか**（顧客ヒアリング）。

> moat の一点突破根拠: 「使うほど賢くなる」が本当に効くのは C（接待オペ適性＝一度使わないと分からない黒子力）。ここは社内記録でしか蓄積できず、食べログにも Places にも無い ＝ 模倣困難。

### 非ゴール（検証 v1 では扱わない）

- 予約の代行・決済（店の予約は人間が行う）
- 全国・全ジャンルの網羅（v1 は東京駅・京橋周辺にフォーカス）
- 完全自動の無人レコメンド（最終判断は人間。ツールは根拠を示して支援する）
- **Google Places 連携 / 自動 enrichment**（検証 v1 は手キュレーションで代替 → §6・§7）
- **接待相手プロフィール / パーソナライズ / 学習ロジック**（記録行動の需要を検証してから → §6）

## 3. ターゲットユーザー

- **一次**: 接待の店を選ぶ営業担当（社会人２・3年目）
- **二次**: チーム全体（蓄積された接待ナレッジを共有・参照する）
- 将来的には他社にも提供するプロダクト（マルチテナント SaaS）を見据えるが、v1 は薄く動く単一ユーザー構成。



## カスタマー　（お金を払う人）

- メイン：単価が高くて、接待メインのお店。普段は稼働率が低い



## 4. 主要ユースケース

1. **探索**: 「キリンの田中部長（日本酒好き・甲殻類NG）を京橋で4名、予算2万〜で接待したい」→ OK / 注意 / NG の理由付きで店がランキング表示される。
2. **NG回避**: キリン接待で「アサヒ専売」の店には NG フラグと根拠が出る。うっかり競合の店を選ぶのを防ぐ。
3. **パーソナライズ**: 同じ相手を再度接待するとき、前回行った店は自動的に避け、相手の好み（日本酒など）に合う店を上位に出す。
4. **ナレッジ蓄積・共有**: 接待後に評価★とシチュエーション（相手の役職・目的・シーン）を記録すると、チームの資産として溜まり、「こういう時はこの店」を次の人が引ける。

## 5. 機能要件

### 5.1 探索（コア）

条件を入力すると、候補店を判定してランキング表示する。

- **入力**: 相手企業 / 接待相手（記録から呼び出し or その場入力）/ **相手の役職** / **会の目的（顔合わせ・御礼・謝罪・慰労…）** / **相手の拠点** / エリア / 人数・個室希望 / 予算 / ジャンル
  - ※ 役職・目的・拠点は 9軸のうち高価値な A(格式)・B(個室の質)・F(シーン適合)・D(動線) を判定するのに必須。現行モックのフォームは材料不足なので拡張する。
- **出力**: 各店に `OK / 注意 / NG` バッジ ＋ 適合スコア ＋ 理由文。NG も除外せず、**ランキング下部に理由付きで残す**（学びになる／人間が最終判断できる）。

### 5.2 NG・適合判定の軸

現行は以下の**粗い4軸**。暗黙知ヒアリングで **最大9軸（status.md の A〜I）に精緻化予定** — どの軸が実際に効くか（重み）と発火条件は、ヒアリングで `[発火条件] × [店属性] → [効果] + [理由文]` の型に落としてから確定する。まずは4軸で骨格を作る:

1. **競合製品NG** — 相手企業がメーカーの場合、競合ブランドを主力に出す店は NG、自社ブランドを扱う店は加点。
2. **格式** — 個室の有無、店のグレード、落ち着き。相手の役職に釣り合うか。
3. **予算** — 1人あたり予算に収まるか。安すぎ／高すぎも判定。
4. **相手の嗜好・NG食材** — アレルギー / 宗教 / 苦手なもの / 好物。

### 5.3 会社ルールの知識化

- **内蔵メーカー辞書**: 企業 → 自社ブランド群 / 競合ブランド群。まずビール4社（キリン / アサヒ / サントリー / サッポロ）を初期内蔵。
- **ユーザー追加ルール**: 任意の企業に「このブランドはNG / 優先」を後から登録できる。

### 5.4 店の飲料系統 enrichment

Google Places に無い「この店は何系のビールか」を補完する。

- **発火条件**: 競合ルールが有効なとき（相手がメーカー）だけ実行。非メーカーならスキップ。
- **対象**: 事前フィルタを通過した少数店だけ（コスト・レイテンシ抑制）。
- **方法**: AI が Web検索で「取扱ビール系統」を推定。
- **確信度**: 『確実 / 推定 / 不明』の3段階で表示し、根拠URLを添える。**不明を不明として扱い、確実に分かる店だけ NG フラグ**を立てる（安全側）。
- **キャッシュ**: 調べた結果は保存して再利用する。

### 5.5 接待相手プロフィール

- 接待相手を「連絡先」として保存・再利用できる（会社・役職・好み・NG食材・メモ）。
- 探索フォームから呼び出すと、その相手の好み・NG が判定に反映される。

### 5.6 接待記録 ＆ 社内ナレッジ

- 接待後に記録を残せる: 相手 / 店 / 日付 / 人数 / シチュエーション（目的・シーン）/ 評価★ / 良かった点メモ。
- 記録は次の探索に反映される:
  - **重複回避**: 同じ相手に前回と同じ店を出さない。
  - **好み学習**: 相手が気に入った傾向を加点。
  - **社内ナレッジ**: 過去に高評価だった店 / 同じシチュエーションで使われた店を、根拠つきで推薦・共有する。

## 6. スコープ

### 検証 v1（真の thin slice ＝ UX と核仮説の検証機）

「薄いを名乗る厚い製品」を避け、**外部 API ゼロ・コスト≒ゼロ**で核仮説を検証する build-to-learn 構成:

- **データ源**: **京橋 20〜30 店を手キュレーション**（JSON/TS seed）。格式(S/A/B) / 個室 / 予算帯 / ビール系統 / 確信度 を **手タグ**。← Places の代替（理由は §7）。
- **画面**: **探索1画面のみ**（条件入力 → 理由付きランキング。合意済みモックを実装）＋ 末尾に **「この店に決めた／後で評価」だけの軽量キャプチャ**。
- **判定**: NG軸4つ（競合製品 / 格式 / 予算 / NG食材）を **確定的ルールエンジン**で。**格式は手タグ S/A/B で確定**（Places 近似はしない）。
- **メーカー辞書**: ビール4社 seed ＋ ユーザー追加ルール。
- **ビール系統**: **手タグ**（自動 enrichment は §8 の spike 合格後に導入）。
- **認証**: 無し or ベタ1本（本格認証は不要）。
- **テーブル**: `venues` + `brand_rules`（+ 記録行動検証用の最小 `records` or Google Sheet）。**guests / パーソナライズ / 学習は作らない**。

**残す良い設計**: NG を除外せず理由つきでランキング下部に残す UX ／ 確信度＋「不明」許容の安全側思想。

### 拡大フェーズ（3ゲート通過後 → §8）

- **Google Places 連携**（データ源の自動化・スケール）。
- **自動 enrichment**（LLM + Web検索でビール系統を推定・キャッシュ・確信度）。
- **接待相手プロフィール / パーソナライズ（重複回避・好み加点）/ 社内ナレッジのスコア反映**。

### 将来（当面作らない）

- マルチテナント（スキーマに余地だけ残す）
- 全国・全ジャンル展開
- ナレッジ検索・閲覧の専用UI
- 予約連携・シチュエーション別テンプレの高度化

## 7. 制約・前提

- **検証 v1 の店データ源は手キュレーション**（京橋 20〜30 店を人手でタグ付け）。Google Places は**拡大フェーズで導入**。
  - 理由: (a) 良い接待店ほど Places の構造化データが薄い/未掲載 (b) price_level 5段階では「予算2万〜」を判定できない (c) API コストと enrichment 精度リスクを UX 検証から切り離す。
- ビール系統は v1 では手タグ。自動 enrichment は Web検索ベースで 100% 精度は出ない前提 → 確信度表示と「不明」許容で担保（§8 spike で確実率を実測してから採用）。
- 最終判断は人間。ツールは根拠を提示する支援役に徹する。

## 8. 成功基準（＝ SaaS へ厚くする前の 3 ゲート）

デモが動くこと（コード成功）ではなく、**核仮説が正しいこと（コンセプト成功）**を測る:

1. **UX ゲート**: 先輩営業に手キュレーション結果を見せ「この順位・この理由、妥当？」→ 根拠が刺さり、新人が納得して選べるか。
2. **再現性ゲート（enrichment spike）**: 京橋 10〜15 店で系統を実測し、`確実` 率 **≥ 40%**。→ ビール自動化を本採用するか。
   - 未満なら: ビール系統を「手タグの参照情報」に降格し、wedge を格式暗黙知＋社内ナレッジへ寄せる。
3. **需要ゲート**: 想定バイヤーに当て「この痛みに金を払うか」。→ 製品化に進むか。

いずれかが No → 8テーブル / Places / 自動化を作る前に方向転換する（これが検証 v1 の主目的）。

## 9. 実装前タスク / オープン論点

### 実装前に回す（検証タスク）

- **【最優先】暗黙知ヒアリング**: センスの良い社員に、店選びの暗黙知を 9軸（status.md A〜I）を叩き台に言語化してもらう。各軸を `[発火条件（相手/会の属性）] × [店属性] → [効果] + [理由文]` の粒度まで。特に確認: どの軸が効くか（重み）／新人がやらかす典型／機械では拾えない基準（格式ランクの付け方・「接待慣れした店」の見分け方）。結果を §5.2 の軸・§5.1 の入力フォームに反映。
- **enrichment feasibility spike**: 京橋 10〜15 店で AI Web検索により系統を手回し推定し、`確実 / 推定 / 不明` の分布を実測（§8-2 のゲート）。lead-harvest 系 WebSearch 資産を流用。
- **顧客ヒアリング**: 想定バイヤー数名に痛みの実在と支払意思を当てる（§8-3）。
- **手キュレーション seed 作成**: 京橋 20〜30 店に 格式(S/A/B)/個室/予算帯/ビール系統/確信度 をタグ付け。

### 運用で判断 / 拡大フェーズで詰める

- enrichment の情報源優先度（**公式サイト・一次情報を優先**。※食べログのスクレイピングは ToS 違反・訴訟リスクありのため源に含めない）。
- Google Places API の課金アカウント（marchon / 個人）— 拡大フェーズで発行。
- 社内ナレッジの共有範囲（記録に「team 共有」の器をいつ持たせるか）。

> 決着済み（旧オープン論点）: 「格式の自動化度合い」→ **検証 v1 は手タグ S/A/B で確定**（Places 近似はしない）。
