# 会食ナビ — spec/design レビュー（忖度なし）

> 対象: `docs/spec.md` / `docs/design.md`（2026-07-08 時点）
> スタンス: 批評ではなく「実装に入る前に潰すべき論点」を重い順に。判断理由つき。

---

## 結論（先に）

- **ドキュメントの完成度は高い。** spec/design 分離・非ゴール明示・NG を残す UX・enrichment のゲート設計は良い。**書類としては通る。**
- **ただし、プロダクトの「一番おいしい所（競合ビール NG）」が、一番検証されてない所と一致している。** ここが崩れると製品の看板が崩れる。
- **v1 のスコープが「薄く動く1枚」を名乗っているが、実体は 8 テーブル・2 外部 API・3 画面。全然薄くない。** 削るべき。
- 一言でいうと: **作る前に「本当にビール NG が価値の中心か」「Places で高級会食店を扱えるか」を先に殺しにいくべき。** 今のまま実装すると、動くけど誰も使わない or 精度が出ないリスクが高い。

## 🔴 致命的（実装前に必ず詰める）

### C1. 看板機能（競合ビール NG）の需要と再現性が両方とも未検証

- **需要側**: 会食相手が「飲料メーカー社員」で、かつ「競合専売店を避けたい」──この積集合はかなり狭い。会食相手の大半はメーカーですらない。看板に据えるには前提の頻度が低すぎる可能性。
- **供給側（もっと深刻）**: 「この店が何系のビールを出すか」を AI Web 検索で当てる、が製品の核。でもこれは (a) Web に載ってないことが多い (b) 時期で変わる (c) 検証しづらい、の三重苦。**「不明」で返ってくる店が 8 割なら、この機能の価値は実質ゼロ + ノイズ。**
- spec は「確信度3段階 + 不明許容で安全側」と書くが、これは *精度が出なかった時の言い訳の設計* であって、*価値が出る根拠* ではない。**期待ヒット率の見積もりが無い。**
- そもそも、キリンの部長を連れて行くベテランは「アサヒ専売バーは避ける」を既に知っている。そして避け方は簡単で「全銘柄置いてる普通の店を選ぶ」だけ。競合専売なんて専門ビアバーくらいで、そこに部長は連れて行かない。**＝ツールが解こうとしてる問題が、現場ではそもそも大問題じゃない可能性。**

**→ アクション**: 実装前に「京橋の実店舗 10〜15 件」で enrichment を手回しして、確実に系統が判定できる店が何割あるか実測する。ここが 3〜4 割を切るなら看板を差し替える（下記 C2）。

### C2. 新人が本当に失敗するのは「格式ミスマッチ」であって、ビール銘柄ではない

- 新人が会食でやらかす典型は「部長の格に対して店が安っぽい/カジュアルすぎる」「個室じゃなかった」「相手の苦手食材を出した」。**＝格式・個室・NG食材の基礎。ここは enrichment 不要で今すぐ効く。**
- ところが spec は「格式」を4軸の #2 に置きながら、design では「price_level・rating・genre で近似 *or* 人手タグ。実データ見て調整」とオープン論点に丸投げ。**一番価値が高くて一番難しい判定が未設計。** 看板のビールは作り込んで、本丸の格式は後回しになっている。
- 提案: **看板を「競合 NG」から「新人が失礼しない基礎（格式×個室×予算×NG食材）を根拠つきで担保」に寄せる。** ビール NG は目玉ではなくスパイス（あれば嬉しい加点）に格下げ。こっちの方が需要が広く、enrichment 依存も外れる。

### C3. Google Places は「高級会食店」と相性が悪い（3/4 軸が崩れる）

- 良い会食店（料亭・高級寿司・会員制）は Places に構造化データが薄い/無い。price_level 欠損・一見さんお断り・そもそも未掲載。**「良い店ほどデータが無い」。** Places で出てくるのはチェーン居酒屋と食べログ中位帯。
- `price_level` は 0〜4 の 5 段階しかない。「予算2万〜」を price_level=4 から判定するのは不可能。**予算軸（4軸の1つ）が Places だけでは機能しない。**
- 個室有無も Places では不正確なことが多い。
- **＝4軸中3軸（格式・予算・個室）が、弱い/欠損しがちな Places フィールドに依存している。** ここは design の「オープン論点」ではなく前提リスク。

**→ アクション**: v1 は Places をやめて **京橋の店を手でキュレーション（20〜30 件、格式・個室・予算帯・ビール系統を手タグ）** する方が、質・速度・コスト全部で勝つ。Places 連携は「UX が価値ある」と分かってから。

### C4. 「使うほど賢くなる社内ナレッジ」が v1 では空箱（＝設計原則違反）

- 差別化として掲げる「会食記録→社内ナレッジのフライホイール」は、複数人×複数年で回る価値。**v1 は単一ユーザー・記録ゼロ・京橋限定なので、初日は完全に空。**
- なのに 8 テーブル中 4 つ（guests / settai_records / recommendations / user_rules）と、パーソナライズ/学習ロジックを v1 で作ろうとしている。
- これは work-os の CLAUDE.md 設計原則 **「複数プロジェクトで必要性が見えてから構造を作る。事前に空き箱を作らない」に真っ向から反する。** 記録スキーマは v1 の体験に一切寄与しないのに、実装・保守コストだけ乗る。
- おまけに「好み学習」も、相手ごとに記録が 1〜3 件しか溜まらないので *学習* は成立しない。実体は「前回の店を避ける + 手入力の好みを引く」だけ。**呼称が誇大。**

**→ アクション**: v1 から guests/records/knowledge を丸ごと落とす。探索 UX が「使いたい」と検証できてから足す。

## 🟡 中程度（方針に関わる）

### M1. コスト見積もりが数字ゼロ
- spec は「コスト抑制」を連呼するが金額が無い。Sora の常道（トークン/無料枠/工数を事前試算）に反する。
- ざっくり: Places Text Search ≈ $32/1000、Details ≈ $17/1000。1 探索で 15〜30 件 Details を引くと **Places だけで 1 探索 ≒ $0.3〜0.5**。さらに 10 店に LLM+Web 検索 enrichment。**1 探索 $1 前後になり得る。** 数回/年しか使わないツールとしては割に合うのか要試算。手キュレーション（C3）ならここがほぼ 0 になる。

### M2. そもそも「人に聞く」に勝てるのか（採用リスク）
- 新人が会食店を選ぶ頻度は低い（初期キャリアで年数回）。高ストレス・低頻度の意思決定で、専用ツールを開いて多項目フォームを埋めるより、**先輩/秘書に聞く方が速くて信頼できる。**
- spec に「なぜツールが現状（人に聞く）に勝つか」の記述が無い。**最大の競合は食べログでも Google でもなく『詳しい人間』。** ここへの回答が無いと、動いても使われない。

### M3. 食べログをスクレイピング源に据えるのは地雷
- design §6/§8 が enrichment 情報源に「食べログ写真・口コミ」を列挙。**食べログのスクレイピングは ToS 違反で、運営は訴訟に積極的。** 将来 marchon の SaaS 化を見据えるならなおさら乗せてはいけない。公式サイト/一次情報に限定を明記すべき。

### M4. 成功基準がデモ通過であって、価値検証じゃない
- §8 成功基準＝「キリンシナリオで NG フラグが出る」。これは *コードが動く* 証明で、*コンセプトが正しい* 証明ではない。
- 欲しいのは「新人が選んだ店を、先輩が『それでいい』と承認する率」のような**採用/的中の基準**。今のままだと「デモは通ったが使われない」を検知できない。

### M5. 「内部ツール」なのか「売るプロダクト」なのか未確定で、全部が変わる
- spec は「v1 は薄い単一ユーザー」と言いつつ「他社提供の SaaS を見据える」「tenant_id の余地」と両にらみ。
- どちらかで作るべきものが真逆になる:
  - **内部ツール** → マルチテナント/ナレッジ基盤は不要、京橋手タグで十分、成功基準は「marchon で使われるか」。
  - **売る製品** → marchon 1 社で検証しても意味が薄い、市場性検証が先、京橋限定では語れない。
- **v1 は「内部ツールとして京橋で1人が使う」に振り切るのを推奨。** SaaS 妄想はスキーマにすら入れない（YAGNI）。

## 🟢 良い点（残すべき判断）

- spec/design/status の3分離と非ゴールの明示（自作フロー通り、良い）。
- **NG を除外せず理由つきでランキング下部に残す** — 学び・人間最終判断の思想が UX に落ちてて良い。ここは維持。
- enrichment を「メーカー時のみ × 事前フィルタ通過店のみ」でゲートするコスト設計（正しい所にコスト意識がある）。
- 確信度 + 不明許容の安全側設計（思想は正しい。問題は C1 の通り“不明だらけ”になった時に価値が残るか）。

## 推奨: v1 をこう削る（thin slice の再定義）

現状の v1 は thin を名乗る thick。**リスクの高い仮説を最速で殺す** 形に組み替える提案:

| 項目 | 現 v1 | 推奨 v1（真の thin slice） |
|---|---|---|
| データ源 | Google Places API | **京橋 20〜30 店を手キュレーション（JSON/seed）** |
| ビール系統 | AI Web 検索 enrichment | **手タグ（自分で 20 店調べて埋める）** |
| 画面 | 探索 + 相手管理 + 記録 | **探索1画面のみ** |
| テーブル | 8 テーブル | **venues + brand_rules の 2 つ** |
| 認証 | iron-session | **無し（or ベタ1本）** |
| 記録/学習/ナレッジ | あり | **全部無し** |
| 外部 API | Places + LLM | **無し（enrichment は後）** |

これで検証できること（＝作る目的）:
1. **「根拠つきランキング」の UX が、実際に新人の意思決定を助けるか。** ← 一番大事な未検証点。
2. **格式×個室×予算×NG食材の基礎判定が、先輩の判断と一致するか。** 先輩に「この順位、妥当？」と見せて答え合わせ。
3. ビール NG が「おっ」と刺さるか、蛇足か（手タグ 20 店で十分体感できる）。

ここが Yes なら → Places 連携（自動化）→ 記録/ナレッジ（フライホイール）へ順に厚くする。**No なら、8 テーブル作る前に方向転換できる。**

## 実装前に answer が欲しい問い（優先順）

1. これは **marchon 内部ツール**か、**売る製品**か。（M5 — これで全部決まる）
2. 看板は **競合ビール NG** のままか、**格式×基礎の担保** に寄せるか。（C1/C2）
3. v1 データは **Places 自動**か **手キュレーション**か。（C3）
4. Places/LLM の **1 探索あたり許容コスト**はいくらか。（M1）
5. enrichment の手回し実測（京橋 10 店）を、実装前にやるか。（C1 — やるべき）
